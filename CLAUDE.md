# プロジェクトの開発ガイドライン

## 基本開発フロー(Maker-Checker アーキテクチャ)
タスクを依頼された場合、以下の役割を明確に分離した自律サイクルを回して完結させてください。

1. **計画 (`/plan`)**
   - 修正や実装を始める前に、必ず `/plan` を実行して対応方針と影響範囲のタスクリストを構築・提示すること。

2. **実装フェーズ (Maker Agent)**
   - 計画確定後、独立したセッションとして `.claude/agents/maker.md` のペルソナ・開発原則を起動/バインドし、コードの実装・修正を行うこと。

3. **検証フェーズ (Checker/Evaluator Agent)**
   - 実装完了後、完全に別の独立したセッションとして `.claude/agents/reviewer.md` を起動し、コードのレビューと自動テストを実行させること。

## 完了・停止条件
- チェッカー（`.claude/agents/reviewer.md`）が「VERDICT: PASS」を出力し、かつすべての自動テストが100%成功するまで、自動的に「プランの見直し ➔ Makerによる修正 ➔ Reviewerによる再検証」のループを繰り返すこと。
- レビュアーが「VERDICT: REJECT」を出力している間は、絶対にタスクを完了（Done）とみなしてはならない。

## 暴走防止ルール（ガードレール）

* **最大ループ回数の制限**:
  - 「プラン ➔ Maker修正 ➔ Reviewer検証」のサイクルは**最大5回**までとしてください。
  - ループを実行するたびに、心の中でカウンターを `+1` し、現在何回目の試行かを意識すること。

* **強制停止条件**:
  - 5回目の検証でも「VERDICT: PASS」にならない場合は、それ以上の自動修正を**直ちに停止（Abort）**すること。
  - 3回連続で全く同じテストエラーが発生し、修正の進捗が見られない場合も、回数に関わらずその時点でループを打ち切ること。

* **人間へのハンドオフ（報告ルール）**:
  - ループが上限に達して停止した場合、勝手にコミットを確定させず、以下の3点を人間にまとめて報告し、指示を仰いでください。
    1. 試みたアプローチの履歴（何をどう直そうとしたか）
    2. 解消できなかった具体的なエラー内容・ログ
    3. AIの視点から見た「なぜ修正がうまくいかないか」の仮説
