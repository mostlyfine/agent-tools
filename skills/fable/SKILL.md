# Tackling Hard Tasks — 分解・検証・次の一手

## Overview

難しいタスクは「単独で検証可能な小さい増分」の連鎖に変換し、増分ごとに **実行 → 証拠 → 判断** のループを回す。進捗は感覚ではなく証拠で測る。

**原則: 実行していないものを「動く」と言わない。証拠が主張に先行する。**

## When to Use

- 影響範囲が3ファイル以上、または公開インターフェースに触れる
- 仕様や挙動に未知の要素が2つ以上ある
- 完了条件を1文で書けない
- 同じ失敗が繰り返されている
- 完了報告の直前（最終チェックとして）

**使わない場面**: 1ファイルの機械的修正、読めば答えが分かる質問。

## 1. 分解 (Decompose)

順番に実行する:

1. **完了条件を1文で書く。** 「何がどう動けば完了か」を、検証コマンドまたは観察方法とセットで定義する。検証方法が書けないなら、タスクをまだ理解していない。
2. **未知と既知を仕分ける。** 不明な仕様・予測できない挙動を列挙し、リスクの高い未知から先に調査で潰す。機械的な作業は後回しでよい。
3. **検証単位で増分に切る。** 各ステップは「単独で検証可能な状態変化」にする。工程単位（調査→実装→テスト）やファイル単位で切らない——最後まで何も検証できない分解は分解ではない。
4. **診断可能なサイズまで割る。** 1ステップは、失敗したとき原因候補が1つに絞れる大きさ。変数を同時に2つ動かさない。
5. **計画をTodoに書き出す。** 頭の中に置かない。計画が変わったらTodoを先に更新する。

## 2. 検証 (Verify)

- **増分ごとに検証する。** 最後にまとめて検証すると、失敗時に原因が特定できない。
- **チェックが失敗しうることを確認する。** 一度も落ちたことのないテストは検証ではない。先に red を見る。
- **コードの見た目ではなく挙動を検証する。** テストが通ることと、実際のフローが動くことは別。可能なら本物の経路を動かして観察する。
- **完了宣言の前に元の依頼文を読み直す。** 要求と成果物を1項目ずつ突き合わせる。作業中に目的がすり替わっていることは珍しくない。

## 3. 次の一手 (Decide)

各ステップの後に必ず問う: **「結果は予想通りか？」**

| 結果 | 次の行動 |
|------|---------|
| 予想通り成功 | 計画の次のステップへ |
| 予想外だが成功 | なぜ動いたか理解してから進む。理解せず進むと後で壊れる |
| 失敗（1回目） | 根本原因を診断してから直す。症状への絆創膏を貼らない |
| 同じ失敗（2回目） | 同じ修正を繰り返さない。仮説そのものを変える |
| 失敗（3回連続） | ステップではなく計画が間違っている。分解からやり直す |
| 新事実で前提が崩れた | 即席で流さない。計画を明示的に書き直してから再開する |
| 不可逆操作・スコープ変更に直面 | 停止して人間に確認する |

## Red Flags — この思考が浮かんだら停止

| 頭に浮かんだ言い訳 | 現実 |
|-------------------|------|
| 「たぶん動くはず」 | 実行して確かめるまで完了ではない |
| 「時間がないのでまとめて検証しよう」 | 失敗時に原因を特定できなくなり、かえって遅くなる |
| 「さっきと同じ修正をもう一度」 | その仮説は死んでいる。診断に戻る |
| 「ついでにこれも直そう」 | スコープ逸脱。計画に追加してから着手する |
| 「このステップは簡単だから検証不要」 | 難しいタスクの中の「簡単なステップ」が一番よく壊れる |
| 「エラーは出ているが本題と無関係だろう」 | 未解決の失敗は複利で効く。無関係だと証明してから進む |

## Common Mistakes

- **工程で分解する**（調査フェーズ→実装フェーズ→テストフェーズ）→ 最後まで検証可能な成果がない。検証単位で縦に切る。
- **検証＝テストが通ること、と考える** → テストは検証の一部。挙動の観察・依頼文との突き合わせを含めて検証。
- **失敗を積み残して先に進む** → 「後で直す」の失敗は、後のステップの失敗診断を汚染する。その場で解消するか、明示的にTodo化して影響範囲を隔離する。
